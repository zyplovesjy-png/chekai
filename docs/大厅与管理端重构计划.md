# 大厅与管理端重构计划

> 目标：独立管理端 + 大厅改版（资料/好友/房间/排行榜）+ 结构化对局入库。  
> 风格：延续茶馆绿金；手机竖屏铺满；管理端可稍宽。  
> 原则：分阶段垂直切片，不破坏现有扯旋规则引擎与 `server/game-rules.test.js`。

## 一、数据库 Schema

### 1.1 `users`（扩展）

| 列 | 类型 | 说明 |
|----|------|------|
| id | INTEGER PK | |
| username | TEXT UNIQUE | |
| nickname | TEXT | |
| password_hash | TEXT | |
| avatar_path | TEXT | |
| role | TEXT | `admin` \| `player`，默认 `player` |
| disabled | INTEGER | 0/1，禁用后不可登录 |
| created_at | INTEGER | unixepoch |

迁移：`ALTER TABLE` 补 `role`/`disabled`；种子写入 `admin`；已有 user1–8 标为 `player`。

### 1.2 好友

**隐式全员好友**：所有 `role=player` 且未禁用的账号互为好友，不建 `friends` 表。  
管理端新建的玩家自动进入该集合。

### 1.3 对局会话 / 手牌

```sql
-- 一场房间对局（开局→结算）
CREATE TABLE game_sessions (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  room_code TEXT NOT NULL,
  room_name TEXT NOT NULL,
  host_username TEXT NOT NULL,
  round_limit INTEGER NOT NULL,
  started_at INTEGER NOT NULL,
  ended_at INTEGER,
  end_reason TEXT
);

-- 每一局（hand）
CREATE TABLE hand_records (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  session_id INTEGER NOT NULL REFERENCES game_sessions(id),
  hand_no INTEGER NOT NULL,
  played_at INTEGER NOT NULL DEFAULT (unixepoch()),
  end_reason TEXT  -- compare | all_folded | rest_cross | ...
);

-- 每局每位玩家
CREATE TABLE hand_players (
  id INTEGER PRIMARY KEY AUTOINCREMENT,
  hand_id INTEGER NOT NULL REFERENCES hand_records(id),
  username TEXT NOT NULL,
  nickname TEXT NOT NULL,
  cards TEXT NOT NULL DEFAULT '',  -- 紧凑编码，见下
  delta INTEGER NOT NULL DEFAULT 0,
  result TEXT NOT NULL DEFAULT 'tie'  -- win | loss | tie
);
```

旧表 `game_records` / `player_records` 保留只读兼容，新写入走上述表。

### 1.4 `player_stats`（替换/扩展 `user_stats`）

| 列 | 说明 |
|----|------|
| username | PK |
| total_hands | 总局数 |
| wins / losses / ties | 按单局 delta 判定 |
| total_profit | 历史净输赢（筹码） |
| updated_at | |

净胜率 = `wins / total_hands`（总局为 0 时视为 0）。

### 1.5 牌紧凑编码

不存前端 SVG。使用引擎已有 `card.id`（如 `rQ1`、`b101`、`joker`），多张用 `|` 连接：

```
rQ1|b101|r3|joker
```

解码：`str.split('|')`。展示时前端用现有 `CardView` / 牌面资源按 id 渲染。

## 二、API 设计

### 2.1 认证 / 用户

| 方法 | 路径 | 说明 |
|------|------|------|
| POST | `/api/login` | 返回 `user.role`；禁用账号拒绝 |
| GET | `/api/user/profile` | 含 role、stats 摘要 |
| PATCH | `/api/user/profile` | 改昵称、密码（需旧密码） |
| POST | `/api/user/avatar` | 已有，复用 |

### 2.2 好友 / 在线 / 概览

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/friends` | 全部玩家：头像、昵称、`online` |
| GET | `/api/lobby/overview` | 在线人数、进行中房间数 |

在线：全局 WS presence（大厅 `presence` + 房间 `join_room` 共用 `onlineUsers` Map）。

### 2.3 房间列表

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/rooms/list` | 自己创建的 + 好友创建的；状态文案 |
| POST | `/api/rooms/create` | 已有 |
| POST | `/api/rooms/join` | 已有；观战：已在房内未入座即可 |

状态示例：`等待中（5人）` / `已开局（5人）第5/16局`。

### 2.4 排行榜

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/leaderboard?type=profit\|record\|winrate` | 实时读 `player_stats` |

- `profit`：按 `total_profit`  
- `record`：按 wins 降序，再 losses  
- `winrate`：按 `wins/total_hands`，总局≥1

### 2.5 对局记录

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/records` | 分页；管理端看全部，玩家看自己参与的 |
| GET | `/api/records/:sessionId` | 会话详情 + 各手牌 + 玩家牌与 delta |

### 2.6 管理端（需 `role=admin`）

| 方法 | 路径 | 说明 |
|------|------|------|
| GET | `/api/admin/users` | 账号列表 |
| POST | `/api/admin/users` | 创建账号 |
| PATCH | `/api/admin/users/:username` | 改昵称/密码/禁用 |
| DELETE | `/api/admin/users/:username` | 删除（不可删 admin 自身） |
| POST | `/api/admin/users/:username/avatar` | 代上传头像 |
| GET | `/api/admin/records` | 全部对局会话列表 |

## 五、结算挂钩点（Phase 3）

1. `POST /api/rooms/:code/start` 成功后 → `createGameSession(room)`  
2. 每手结束（`logRoundEnd` 同路径：compare / all_folded / rest_cross）→ `saveHandRecord(room, endReason)`  
3. `endGameAndSendSettlement` → `finishGameSession` + 会话级汇总（可选）；单局统计已在 2 更新  

`saveRoundResults` 旧逻辑改为写新表或废弃，避免双计。
