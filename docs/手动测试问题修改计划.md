# 扯旋 手动测试问题修改计划（交接给执行 AI）

> 依据：`docs/结算规则与改造方案.md`（唯一事实来源）、`docs/游戏术语表.md`
> 现状基线：`node server/game-rules.test.js` 全部 15 项通过——**结算算钱的核心引擎是正确的**，本轮问题集中在下注交互、局间流转、座位动态化与 UI 展示层。
> 术语：喊价 = `player.committed`（整局累计），底池 = `state.potPi`（只含底注+芒果+弃牌支付）。

## 〇、已与产品方对齐的口径（本次新确认）

1. **第一轮允许"休"**：庄家下家先发话、本轮尚无人抬价时，玩家可以"休"（符合文档 2.4「本轮没人抬价前都可以休」）。当前前端把第一轮强制当作已下注，需放开。
2. **下注滑条数字 = 整局累计喊价目标值**：例如已喊 30、拖到 60，表示"整局共喊 60"，不是"再加 60"。保持与后端 `committed` 累计语义一致，不做换算。UI 文案可优化但语义不变。

## 一、必须修复的 Bug（P0，与文档明确冲突）

### P0-1 第一轮无法"休"，且"瞧"会报错

- **位置**：`client/src/pages/room/useRoomGameController.ts` 第 436 行
  ```ts
  const betStarted = game.betStarted || game.betRound === 1;
  ```
- **根因**：`|| game.betRound === 1` 使第一轮恒等于"已有人抬价"，`ActionBar` 只渲染 `敲/弃/返/瞧` 面板，没有"休"按钮；而此时 `currentBet=0`，点"瞧"会被 `doSee`（`currentBet<=0` 返回 null）拒绝并报错。
- **改法**：改为直接透传后端语义：
  ```ts
  const betStarted = game.betStarted;
  ```
  这样第一轮开局（`betStarted=false`）显示 `休/弃 | 叫分 | 敲/叫` 面板，与文档 2.4 一致；后端 `doRest` 本就允许（`canRestNow()` 判断 `!betStarted && currentBet<=0`）。
- **验收**：第一轮庄家下家可点"休"；无人抬价时"瞧"不出现或不报错；有人抬价后"休"消失、出现"瞧/返"。

### P0-2 两人局一人输光后不弹窗，直接结束整场

- **位置**：`server/server.js` `startNextRoundAuto`（约第 257–272 行）；相关：`server/game.js` `doCompare` 末尾把 `pot<=0` 标记 `eliminated=true`（约第 1178 行）。
- **根因**：`doCompare` 结算后已把破产者标记 `eliminated`。而两人局检测用
  ```js
  const seatedAlive = engine.players.filter(p => !p.eliminated);
  const broke = seatedAlive.filter(p => p.pot <= 0);
  if (seatedAlive.length === 2 && broke.length === 1) { ...弹窗... }
  ```
  破产者已被 `eliminated` 过滤掉，`seatedAlive.length===2 && broke===1` 永不成立，流程落到"自动离座 → 剩不足 2 人 → `endGameAndSendSettlement`"，弹窗（文档 3.4）永远不触发。
- **改法**（二选一，推荐 A）：
  - **A**：两人局判定改为基于"仍占座 + 本局是否破产"，不依赖 `eliminated`。在应用 `pendingBuyIn` 之后、自动离座之前，用"仍在座位上的玩家数 == 2 且其中恰有 1 人 `pot<=0 && pendingBuyIn<=0`"作为触发条件；命中则设置 `room.awaitingBuyInDecision` 并 `return`，**不要**先把破产者 `eliminated`/离座。
  - **B**：在 `doCompare` 里不直接 `eliminated`，把"破产→淘汰"的决定权全部上移到 `startNextRoundAuto`，由它统一判断两人局特殊分支后再决定淘汰/离座。
- **注意**：`buyin_decision` 的 `continue` 分支已实现（`server/server.js` 约 583–599 行）；确认破产者 `eliminated=false` 会在加簸后恢复。`settle` 分支走 `endGameAndSendSettlement(room,'buyin_exit')`，确认结算金额守恒（局间 `committed` 已清零，`finalPot = pot + pendingBuyIn` 正确）。
- **验收**：两人对局中一方输光 → 出现「加簸继续 / 立即结算退出」弹窗；选加簸后下一局正常开始；选结算走总结算且各人 delta 之和为 0。

### P0-3 中途入座会打乱"当前局"的发话/发牌顺序

- **位置**：`server/rooms.js` `sitDown` 的 midGame 新入座分支（约第 156 行调用 `engine.rebuildPlayersFromSeats(room.seats)`）。
- **根因**：新观战者坐下时立即 `rebuildPlayersFromSeats()`，把新玩家插入 `players` 数组、改变现有玩家的物理索引。本局后续 `dealThirdCard/dealFourthCard` 依赖 `activeIndicesCCW`（物理顺序）推导发话/发牌次序，可能因此错乱。违反文档 4.3「入座不影响当前局，下局才加入」。
- **改法**：中途"新入座"时**不要**立即 `rebuildPlayersFromSeats`。仅写入 `room.seats`（带 `joiningNextRound=true`）与 `initialBuyIns`，让 `startNewRound` 在下一局开始时通过 `alivePlayers()` 自然纳入。`rebuildPlayersFromSeats` 只保留给"局间换座"和"局间起身"场景调用（那些场景本就不影响进行中的对局）。
- **验收**：对局进行中有人坐下 → 本局的发话顺序、发牌、opener 判定完全不受影响；该玩家在座位上观战，下一局正式参与、按新座位物理顺序排序。

## 二、需核对并统一的问题（P1）

### P1-1 后端 raise 对"同轮再加注"多余地强制 ≥ minBet

- **位置**：`server/server.js` `player_action` 的 `raise` 分支（约第 653 行）`if (betAmount < s.minBet) ...`。
- **规则**：文档 2.3「同一轮内的再加注：只需高于当前最高喊价即可，无翻倍等幅度要求」。引擎 `doRaise` 已正确（只在 `!s.betStarted` 时要求 ≥ minBet）。
- **改法**：把 `raise` 分支的 `betAmount < s.minBet` 校验改为仅在 `!s.betStarted` 时生效（与引擎一致）。实战中 `betStarted` 后 `currentBet≥minBet` 一般不触发，但口径应统一，避免后续误判。

### P1-2 `see`/`raise`/`call` 的额度校验术语统一

- **位置**：`server/server.js` `player_action`（约 646/654/663 行）用 `p.roundCommitted`；引擎用 `p.committed`。
- **现状**：`resetBettingRound` 里 `roundCommitted = committed`，二者恒等，暂无 bug。
- **改法**：统一改用 `p.committed`（累计口径），并可考虑逐步移除 `roundCommitted` 字段（前端 `ActionBar.playerRoundCommitted`、`gameStore` 一并收敛），减少语义混淆。若不移除，至少在注释中标注"= committed（整局累计）"。

### P1-3 弃牌局/全员休局不生成对局历史记录

- **位置**：`client/src/pages/room/useRoomGameController.ts` 第 146 行
  ```ts
  if (game.phase === 'done' && game.compareResult && game.round !== capturedRound.current) { ... }
  ```
- **根因**：`all_folded`、`rest_cross` 走 `handleBettingDone`，不产生 `compareResult`，因此这两类局不会被写入 `roundHistory`，历史回看会缺局。
- **改法**：后端在 `all_folded`/`rest_cross` 结束时也构造一个轻量 `compareResult`（含 winner 或 `reason`、各玩家 `lastDelta`），或前端放宽记录条件为"`phase==='done'` 即记录，`compareResult` 可空"。保证每局都有历史条目。
- **验收**：一局所有人弃到只剩一人 / 全员休作废后，历史里都能查到该局及其结果。

## 三、UI / 展示层（P1，用户实测点名的"下注按钮 / 顺序 / 结算数字 / 界面"）

> 引擎数值正确的前提下，重点排查以下展示是否与 2.6 / 2.2 一致。
> **2026-07-09 补充**：以下 4 项已通过 Playwright 浏览器自动化 + 截图视觉核对确认（截图见 `output/playwright/`）。

### UI-5 喊价旁多显示一个圆圈（chip-dot 残留）【用户新反馈，已确认】

- **现象**：玩家下注后，头像旁出现「彩色圆圈 + 数字」，像是两套筹码 UI 叠在一起。
- **位置**：
  - `client/src/pages/room/components/PlayerSeat.tsx` 第 79–87 行（自己座位）、第 113–121 行（他人座位）
  - `client/src/styles/room.css` 第 1771–1784 行 vs 第 2264–2283 行（样式互相覆盖）
- **根因**：
  1. `self-chip-stack` / `seat-stack-row` 已用 `chip-dot` 圆圈展示**簸簸**（`player.pot`）；
  2. `seat-bet-chip` 又用第二个 `chip-dot` 展示**喊价**（`player.committed`）；
  3. CSS 在 `.room-page.new-table` 下把 `seat-bet-chip .chip-dot` 重新设为 `display: inline-block`（2283 行），覆盖了旧主题里的 `display: none`（1784 行），导致喊价也带圆圈。
- **改法**：
  - **喊价只显示纯数字**，去掉 `seat-bet-chip` 内的 `<span className="chip-dot" />`；
  - 喊价标签改用「喊价 30」或单独小字样式，与簸簸的圆圈筹码图标区分开；
  - 清理 `room.css` 中 `seat-bet-chip .chip-dot` 的重复/冲突规则，只保留一套。
- **验收**：下注后座位旁只有**一个**圆圈（簸簸）+ **一个**纯数字（喊价），不出现双圆圈。

### UI-6 簸簸数随喊价减少，容易误解【用户新反馈，已确认】

- **现象**：玩家喊价后，头像旁簸簸数字立刻减少（例如 100 → 90），像是「簸簸被扣掉了」。
- **位置**：`PlayerSeat.tsx` 显示 `player.pot`；引擎 `game.js` `raiseCommitment` 从 `pot` 预扣到 `committed`。
- **根因**：后端语义是「喊价预扣」：`pot` = 手中剩余，`committed` = 已喊累计。UI 只展示 `pot` 作为簸簸，没有体现「总簸簸 = pot + committed」。
- **规则口径**：文档 2.2 示例中底注会从簸簸扣除（合理），但喊价期间玩家更自然的理解是「簸簸总数不变，旁边另显示喊了多少」。
- **改法**（推荐，只改展示不改引擎）：
  - 座位簸簸显示改为 `player.pot + (player.committed || 0)`（总簸簸）；
  - 喊价单独显示 `player.committed`（纯数字，见 UI-5）；
  - `ActionBar` 滑条 `sliderMax` 已用 `playerChips + playerRoundCommitted`，与上述一致。
- **验收**：喊价 10 后，总簸簸仍显示 100，喊价旁显示 10；弃牌/比牌结算后总簸簸才按结果变化。

### UI-7 对局记录点击无反应【用户新反馈，已 Playwright 确认】

- **现象**：点右上角「对局记录」，界面无任何变化。
- **位置**：`client/src/pages/RoomPage.tsx` 第 141–196 行。
- **根因（两层）**：
  1. **空列表不渲染面板**：`onClick` 会切换 `showHistory`，但面板条件为 `showHistory && game.roundHistory.length > 0`——记录为 0 时面板完全不出现，用户感觉按钮坏了；
  2. **有比牌的局也不一定能记入**：`useRoomGameController.ts` 第 146 行要求 `compareResult` 存在才 `addRoundHistory`；弃牌局/全员休局不产生 `compareResult`（见 P1-3），导致玩了几局后 `roundHistory` 仍可能为空。
- **改法**：
  - `showHistory === true` 时**始终**渲染面板；`length === 0` 显示「暂无对局记录」空态 + 关闭按钮；
  - 配合 P1-3 修复记录写入，确保每局结束都有条目；
  - 可选：按钮旁数字 `roundHistory.length` 为 0 时点击也给 toast 提示。
- **验收**：无记录时点击弹出空态面板；有记录时正常展示；弃牌局/休芒局也能查到。

### UI-8 「返」按钮与滑动条重叠【用户新反馈，已 Playwright 确认】

- **现象**：轮到自己操作时，右侧「返」圆形按钮与竖向滑动条区域重叠，难以点击。

  - 将 `bet-turn-panel` 改为 `display: grid; grid-template-columns: 1fr auto 1fr;` 三列：左按钮组 | 滑条 | 右按钮组；
